# Frontend as native-ESM modules, no build step

**Decision:** The dashboard is split from one 1224-line inline `public/index.html` into a thin HTML shell + `public/css/app.css` + a `public/js/*` module tree, loaded as **native ES modules** (`<script type="module">`) straight from disk — no bundler, no transpile, no new deps. A leaf module `app-events.js` holds the shared mutable state (`lastState`/`inFlight`) and registerable `onRender`/`onAuthError` callback slots; `main.js` is the only orchestrator and registers those callbacks at bootstrap, so lower modules call up through `app-events.js` and never import `main.js` — keeping the import graph a DAG. A nested `public/package.json` `{"type":"module"}` makes `node:test` treat the modules as ESM.

**Why:** The single file had outgrown maintainability, but a build step would break the project's no-build invariant (source on disk = running code; see [node-builtins-only](decisions/node-builtins-only.md)). Browsers run ESM natively, so modules buy separation of concerns and unit-testable pure helpers (`grouping`, `selection`) for free. The `app-events` leaf exists purely to break the cycle that would otherwise form — every module needs to trigger a re-render or signal auth failure, which lives in `main.js`; routing those through a dependency-free leaf avoids that import-back.

**Rejected:** A bundler (Vite/esbuild) or framework — reintroduces the build step and a dependency tree, for a single-user LAN UI. Letting modules import `main.js` directly — creates an import cycle (`main → api → main`) that ESM tolerates unevenly and that obscures the load order. Keeping the one big file — no longer maintainable at 1200+ lines.

**Constraints it creates:** The import graph must stay a DAG — nothing imports `main.js`; cross-module render/auth signals go through `app-events.js`. New frontend assets must be served by `serveStatic` under `/js/` or `/css/` (added before the `/api/` auth gate, unauthenticated like `/vendor/*`). Keep the no-build invariant: every `public/js/*` file must run directly in the browser as ESM. Per-module detail is documented in `public/js/README.md`; keep it there, not in `.ai/`.
