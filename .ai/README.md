# `.ai/` — Project Knowledge Base

The source of truth for project knowledge (see CLAUDE.md). Agents read this
**before** searching raw source, so the project doesn't pay to re-derive its own
structure on every research pass.

## The rule: decision-oriented, not a wiki

> `.ai/` captures **decisions, constraints, invariants, and the map of where
> things live** — the non-obvious *why*. It does NOT restate what the code does.
> If a doc could be regenerated by reading the code, it doesn't belong here.

When in doubt, apply the test: *"Could a competent reader reconstruct this by
reading the code?"* If yes, it's a wiki entry — leave it out. If it captures a
choice, a constraint, or a pointer that saves a search, it belongs.

## Layout

| File / dir        | Holds                                                            |
| ----------------- | --------------------------------------------------------------- |
| `index.md`        | The map. Module → one-line responsibility → path → linked docs. **Read first.** |
| `architecture.md` | High-level system shape: boundaries, package layout, data flow. |
| `decisions/`      | One decision per file: the why + what was rejected.             |
| `patterns/`       | Reusable patterns — added only on the 2nd real use, never speculatively. |

## Maintaining it

Don't edit `.ai/` ad hoc. Use the `sync-knowledge` skill — it owns the format,
the index-update protocol, and the anti-wiki rules. Every PRD declares its
`## Knowledge Base Impact`; `execute-prd` syncs `.ai/` at closeout.

This scaffold ships empty on purpose. Populate it as the project grows — prefer
**editing or retiring** stale entries over appending new ones.

## Formats

The fixed shapes every `.ai/` file follows. The `sync-knowledge` skill owns these
— these mirror it; do not diverge.

### `index.md` router table

One row per module/package. Columns, in order:

`Module | Responsibility (one line) | Path | Related docs`

- **Module** — package/module name.
- **Responsibility (one line)** — its single responsibility, terse.
- **Path** — where it lives.
- **Related docs** — links to any `decisions/` or `patterns/` doc about it.

### `architecture.md` headings

Three required headings, each holding only its concern:

- **System shape** — top-level decomposition: what each package owns and the boundaries between them.
- **Dependency direction** — which packages may depend on which, and the one-way rule preventing cycles.
- **Data flow** — the main flows (request → service → store, etc.).

### `decisions/<slug>.md` template

Byte-compatible with sync-knowledge's decision-record shape:

```markdown
# <Decision title>

**Decision:** <what was chosen, one or two sentences>

**Why:** <the non-obvious reasoning — constraints, trade-offs that forced it>

**Rejected:** <the alternatives considered and why they lost>

**Constraints it creates:** <invariants future code must respect, if any>
```

### `patterns/<slug>.md` template

Added only on a pattern's **2nd real use**, never speculatively.

```markdown
# Title

**Pattern:** <the reusable shape>

**When:** <the situations it applies to>

**Shape:** <the concrete structure to follow>
```
